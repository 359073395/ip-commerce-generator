# 私有知识库管理设计

## 1. 目标

为 IP 商业内容 Agent 建立一套可持续补充、可审核、可追溯、升级不丢失的私有知识库。

核心目标：

- 程序代码和专业知识物理分离，升级代码不覆盖知识。
- 管理员可以通过网页粘贴内容或上传文件补充知识。
- 用户反馈先只在自己的项目内学习，不能污染其他项目。
- 优秀项目经验必须经过管理员审核，才能进入全局私有知识库。
- 上传文件只用于临时提炼，长期只保存结构化知识，控制 VPS 空间。
- 生成结果必须真实命中专业知识，私有库不可用时不得静默降级成低质量结果。

## 2. 范围

本阶段面向单台 VPS，优先保证稳定、简单和可恢复，并保留未来迁移 PostgreSQL 或多节点部署的能力。

包含：

- 外部私有 SQLite 知识库。
- 管理员知识库管理后台。
- 管理员主动补充与审核发布流程。
- 用户项目学习和全局候选知识流程。
- 结构化检索、权限隔离、版本和审计。
- 自动备份、升级保留、迁移和恢复测试。

不包含：

- 多 VPS 实时共享知识库。
- 向量数据库。
- 长期保存原始 Word、PDF、Excel、音视频文件。
- 未经管理员审核的用户经验自动进入全局库。

## 3. 总体架构

### 3.1 存储边界

- `/opt/ip-commerce-generator/data/app.db`
  - 用户、项目、会话、生成历史、任务和实验数据。
- `/opt/ip-commerce-private/knowledge.db`
  - 全局方法卡、项目记忆、候选知识、历史版本和审计记录。
- `/opt/ip-commerce-private/tmp/`
  - 上传文件的临时解析目录。
- `/opt/ip-commerce-private/backups/`
  - 私有知识库自动备份。
- 项目内 `knowledge/`
  - 只保留程序启动、脱敏测试和基础框架所需的公开知识。

应用通过 `KNOWLEDGE_DB_PATH` 获取私有数据库路径，VPS 默认值为：

```text
/opt/ip-commerce-private/knowledge.db
```

现有业务数据库与私有知识数据库保持独立，避免知识迁移影响用户、登录和任务数据。

### 3.2 知识层级

生成时按以下顺序检索：

1. 当前用户、当前项目已经验证的项目记忆。
2. 管理员发布的全局私有方法库。
3. 程序内置的脱敏基础知识框架。

私有库不可读取、损坏或未完成迁移时，服务端返回明确的知识库错误，不允许伪装成完整专业结果。

## 4. 数据模型

### 4.1 全局方法卡

每张方法卡至少包含：

- 唯一 ID、标题、摘要和正文。
- 分类、模块、行业、账号阶段和目标。
- 适用条件、禁用条件、执行步骤和示例。
- 关键词、用户痛点、证明类型和转化方式。
- 来源类型、证据说明、质量分和置信度。
- 状态：草稿、待审核、已发布、已停用。
- 当前版本、创建人、审核人和时间。

只有“已发布”状态可以参与全局检索。

### 4.2 项目记忆

项目记忆必须绑定：

- `userId`
- `projectId`
- 来源生成记录或实验记录
- 用户采用、修改、评分和发布结果
- 适用模块与偏好
- 质量和有效性信号

任何查询必须同时校验用户和项目归属，禁止跨用户、跨项目检索。

### 4.3 候选知识

候选知识来源包括：

- 管理员上传资料的 AI 提炼结果。
- 项目中有明确发布、咨询、线索或成交效果的经验。
- 用户对生成结果的大幅有效修改。

候选知识在进入审核区前需要：

- 去除用户身份、联系方式和客户隐私。
- 提炼为可复用方法，不保存客户原始内容。
- 标记来源类型和验证数据。
- 执行去重、冲突检测和质量评分。

### 4.4 版本与审计

发布、编辑、停用、合并和恢复均生成版本记录。审计记录至少保存操作人、操作类型、时间、修改摘要和前后版本 ID。

## 5. 管理员补充流程

1. 管理员粘贴文字或上传支持的文件。
2. 文件进入临时目录并执行格式、大小和安全校验。
3. 系统提取文本，调用模型生成结构化方法卡。
4. 服务端执行字段校验、去重、冲突检测和质量评分。
5. 管理员预览原始摘要、方法卡、适用范围和风险提示。
6. 管理员可以编辑、合并、驳回或确认发布。
7. 确认发布后写入全局私有库并更新检索缓存。
8. 提炼成功后立即删除原文件；失败文件最多保留 24 小时。

原文件不作为长期知识资产，永久资产是经过审核的结构化方法卡。

## 6. 用户反馈学习流程

普通用户可以提供：

- 结果是否有用。
- 用户修改后的最终版本。
- 是否采用或发布。
- 播放、完播、评论、私信、电话、线索和成交数据。
- 对错误偏好或项目记忆的删除请求。

系统处理规则：

1. 反馈先写入当前用户和项目的项目记忆。
2. 项目记忆只影响该项目后续生成。
3. 有明确效果证据或高价值修正的经验进入候选知识池。
4. 候选内容先匿名化并提炼为通用方法。
5. 管理员审核通过后，才进入全局私有知识库。

## 7. 检索与质量控制

检索查询由以下信息组成：

- 当前功能模块。
- 用户表单和选择项。
- 项目档案和项目历史偏好。
- 行业、账号阶段、内容目标和承接方式。
- 已验证的发布与成交信号。

第一版采用结构化字段、关键词、范围匹配和效果权重，不引入向量数据库。

排序优先考虑：

- 用户与项目精确匹配。
- 模块、行业、阶段和目标匹配。
- 真实效果数据。
- 管理员质量评分和审核状态。
- 适用条件与当前表单事实的一致性。

规则：

- 草稿、待审核和已停用知识不得参与生成。
- 冲突方法根据适用条件选择，不直接拼接。
- 每次生成保存命中的方法卡 ID 和版本，支持追溯。
- 发布、修改或停用后立即生效，不要求重启服务。
- 质量评估需要检查结果是否体现命中方法，而不是只检查篇幅。

## 8. 管理界面

管理员顶部增加“知识库管理”，包含：

- 知识概览：数量、模块覆盖、质量和近期变化。
- 添加知识：粘贴文字或上传文件。
- 待审核：管理员资料和项目候选经验。
- 全局知识：搜索、筛选、编辑、停用、合并和版本恢复。
- 项目学习：按用户和项目查看、纠正或删除项目记忆。
- 备份状态：查看、下载和恢复备份。

普通用户不进入复杂管理后台，只在结果和实验流程中提交反馈，并可以删除本项目已经学到的错误偏好。

## 9. 安全、备份与升级

- 私有目录和数据库只允许应用服务账号与管理员读取。
- 私有数据库不能通过静态文件或公开 URL 访问。
- 上传文件限制类型、大小和字段长度。
- 临时文件提炼成功后立即删除，失败文件 24 小时后自动清理。
- 每天自动备份，保留 7 份日备份和 4 份周备份。
- 管理员可以下载备份，避免同一 VPS 磁盘故障导致全部丢失。
- 每次程序升级和知识数据库迁移前自动创建备份。
- 数据库迁移在事务中执行，失败后恢复旧库并继续运行旧版本。
- 升级器只替换 `/opt/ip-commerce-generator`，不得移动或覆盖 `/opt/ip-commerce-private`。

## 10. 首次迁移

1. 备份现有 `knowledge/`、完整本地知识库和运行数据。
2. 创建独立私有知识数据库和目录权限。
3. 导入现有 handbooks、结构化知识块和验证案例。
4. 对比导入前后的知识数量、分类覆盖和内容校验值。
5. 运行检索、脏数据、权限、生成和浏览器测试。
6. 验证通过后切换运行时检索到私有数据库。
7. 保留可回滚的旧知识包，完成观察期后再清理。

当前专业知识已经进入公开 Git 历史。私有迁移验证完成后，需要创建干净仓库历史或新的纯程序仓库，普通删除文件不能撤回历史中的公开内容。

## 11. 错误处理

- 文件不支持：拒绝解析并给出可接受格式。
- 提炼失败：保留临时文件和错误状态，允许重试或删除。
- 方法卡校验失败：保持草稿，不允许发布。
- 知识冲突：展示冲突方法及适用条件，要求管理员选择、合并或保留并存。
- 私有库不可用：停止专业生成并显示知识库修复提示。
- 备份失败：阻止数据库迁移和高风险升级。
- 恢复失败：保持当前库不变并记录管理员审计事件。

## 12. 测试与验收

必须覆盖：

- 上传、提炼、预览、编辑和审核发布。
- 提炼成功后的原文件删除与失败文件清理。
- 去重、冲突检测和版本恢复。
- 用户、项目和管理员权限隔离。
- 用户反馈只进入当前项目。
- 候选知识未经审核不能全局生效。
- 全局知识发布后能够被正确检索并引用。
- 私有库损坏、缺失和恢复流程。
- 升级前备份、升级保留和失败回滚。
- 脏数据、超长输入、错误格式和模型异常。
- 桌面与手机端完整浏览器流程和控制台健康。
- 现有生成、任务、用户和项目测试不得回归。

验收标准：

- 程序升级后知识数量和生成质量不下降。
- 用户数据不能跨项目或跨用户泄漏。
- 管理员补充知识只需“添加、AI 提炼、审核发布”。
- 私有知识不进入公共 Git 新提交。
- 备份可以在一台干净 VPS 上成功恢复。
