# IP商业方案生成器设计方案

日期：2026-07-06

## 目标

构建一个本地开发、可部署到 VPS 的 Web 应用。应用形态参考灵光「短视频文案创作」闪应用：深色工作台、左侧应用栏、内部模块导航、右侧模块内容、模块内可逐级展开选择，最终调用兼容 OpenAI API 格式的大模型接口生成结果。

产品定位不是通用聊天工具，而是把本地 `ip-commerce-knowledge-base` 知识库产品化。用户输入行业、人设、产品、目标客户、证明材料、成交入口等信息后，后台根据任务类型读取项目内知识库文件，组装结构化提示词，再调用大模型生成 IP 商业方案、内容矩阵、选题、脚本、拍摄清单、承接路径和复盘建议。

## 已确认决策

- 方向：B. IP商业方案生成器。
- 视觉体验：尽量贴近原站的深色工作台和逐级选择体验。
- 交互硬约束：保留「点开继续选择」的层级模块，不做所有选项平铺的大表单。
- 命名硬约束：除外层应用栏外，内部工具选项名称尽量和原版一致。
- 外层应用栏：截图中的「新对话 / 我的收藏 / 我的创作 / 创建闪应用 / 用户区」属于外层应用栏，不要求完全照搬。
- 内部模块名：按知识库顺序新增「IP定位」作为首个模块；后续使用原版「爆款选题、成交选题、痛点选题、脚本创作、文案二创、爆款拆解、文案洗稿」，并新增「带货」模块。
- 完整骨架硬约束：以后任何模块都默认设计完整骨架，不再默认最小骨架。完整骨架必须包含输入表单、层级选项、知识库映射、prompt 组装规则、输出结构、操作按钮、状态处理和错误提示。
- 后端模型：兼容 OpenAI API 格式，支持配置 `OPENAI_BASE_URL`、`OPENAI_API_KEY`、`OPENAI_MODEL`。
- 知识库部署：使用版本化文件知识库，随项目一起部署到 VPS，不依赖 Windows 本地路径。
- 品牌边界：参考原站布局和交互，但不复制灵光 logo、品牌资产、专有图标、源码或文案。

## 用户流程

1. 用户进入工作台。
2. 用户可看到外层应用栏。外层应用栏可参考原站布局，但不要求完全照搬。
3. 用户在内部模块导航选择模块：
   - IP定位
   - 爆款选题
   - 成交选题
   - 痛点选题
   - 脚本创作
   - 文案二创
   - 爆款拆解
   - 文案洗稿
   - 带货
4. 右侧展示当前模块。
5. 在模块内继续选择大类和子类型。
6. 用户填写必要输入。
7. 点击生成。
8. 后台判断任务类型并读取知识库。
9. 后台组装 prompt，调用兼容 OpenAI API 的模型。
10. 前端按模块展示结构化结果。
11. 用户复制、导出或继续切换模块优化。

## 核心交互

### 外层应用栏与内部模块导航

界面保留原站的两层工作台感：

1. 外层应用栏：参考截图中的「新对话 / 我的收藏 / 我的创作 / 创建闪应用 / 用户区」。这一栏可以使用自有品牌和文案，不强制完全照搬原站。
2. 内部模块导航：进入工具后的功能选项必须尽量沿用原版名称。

内部模块导航首版模块：

- IP定位
- 爆款选题
- 成交选题
- 痛点选题
- 脚本创作
- 文案二创
- 爆款拆解
- 文案洗稿
- 带货

### 全模块完整骨架

以下完整骨架是首版实现标准。所有模块都必须按完整结构实现，不允许只做少量字段或单按钮生成。

每个模块的通用结构：

- 模块说明：一句话说明模块用途和适用场景。
- 上下文继承：默认读取「IP定位」结果；用户也可在当前模块补充或覆盖输入。
- 输入区：必填项、选填项、高级项分组展示。
- 层级选项：保留原站「点开继续选择」体验，大类可展开子类。
- 知识库绑定：明确读取个人IP、带货视频或 combined 知识片段。
- Prompt 模板：由模块任务、知识片段、用户输入、输出 schema、合规提醒组成。
- 生成结果：按结构化区块展示，不使用纯聊天气泡。
- 操作按钮：生成、重新生成、复制、导出 Markdown、清空、继承 IP定位。
- 状态处理：空状态、加载中、生成成功、模型错误、API未配置、知识库缺失、输入不足。
- 风险提醒：涉及功效、价格、资质、医疗健康、金融法律等内容时提示人工核验。

IP定位完整骨架：

- 输入分组：基础信息、商业目标、目标用户、人设资产、内容条件、承接与合规。
- 知识库：个人IP核心公式、商业定位、目标用户、人设资产、内容矩阵、默认交付包。
- 输出：信息判断、IP定位一句话、商业路径、目标用户画像、人设资产、内容矩阵方向、待确认项。
- 特殊状态：允许用户先填写必填字段生成初版，再逐步补充高级字段刷新定位。

爆款选题完整骨架：

- 输入：继承IP定位、目标人群、具体场景、核心痛点、情绪方向、想要的平台风格。
- 层级选项：成本类、人群类、奇葩类、头牌类、怀旧类、反差类、最差类、荷尔蒙类。
- 知识库：个人IP选题系统、人群 × 场景 × 痛点/情绪 × 爆款元素公式。
- 输出：每类选题标题、适用人群、第一秒钩子、拍摄角度、推荐脚本类型、风险提醒。

成交选题完整骨架：

- 输入：继承IP定位、产品/服务、客单价、成交入口、信任证明、主要顾虑。
- 层级选项：咨询成交、私域成交、直播成交、到店成交、课程/服务成交。
- 知识库：个人IP商业定位、目标用户、信任资产、咨询/私域/直播承接。
- 输出：成交型选题、成交理由、信任证明使用方式、CTA、承接路径、适合发布阶段。

痛点选题完整骨架：

- 输入：行业背景、目标客户、当前阶段、具体场景、痛点、顾虑、理想结果。
- 层级选项：焦虑痛点、成本痛点、风险痛点、决策痛点、面子/情绪痛点、使用场景痛点。
- 知识库：目标用户画像、需求场景、购买冲突、痛点和潜在需求。
- 输出：痛点清单、场景化选题、情绪钩子、用户原话模拟、对应脚本方向。

脚本创作完整骨架：

- 输入：继承IP定位、选题、目标用户、脚本类型、视频时长、CTA、拍摄条件。
- 层级选项：个人IP脚本、成交脚本、带货脚本；继续展开教知识、晒过程、讲故事、聊观点、种草、测评、过程、故事、直播预热。
- 知识库：个人IP四类脚本卡、带货成交链路、内容类型脚本结构、拍摄剪辑原则。
- 输出：标题、封面文案、黄金3秒、完整口播、分镜/B-roll、字幕重点、CTA、拍摄剪辑清单。

文案二创完整骨架：

- 输入：原文案、目标平台、保留点、规避点、目标人群、想要风格。
- 层级选项：换人群、换场景、换痛点、换情绪、换结构、换平台、本土化表达。
- 知识库：选题系统、脚本结构、差异化表达、平台表达方式。
- 输出：二创版本、结构变化说明、标题/封面文案、口播优化建议、风险提醒。

爆款拆解完整骨架：

- 输入：短视频链接、爆款文案、截图/OCR文本、行业、人设、产品信息。
- 层级选项：钩子拆解、结构拆解、情绪拆解、成交链路拆解、拍摄剪辑拆解、可复用模板。
- 知识库：爆款结构、黄金3秒、情绪刺点、成交链路、商品视觉化。
- 输出：拆解报告、爆款元素、可复用结构、适合本账号的改写方向、下一条选题建议。

文案洗稿完整骨架：

- 输入：原文案、希望保留的信息、必须避开的表达、目标行业、目标人群、风格。
- 层级选项：结构重组、痛点重写、观点重写、故事重写、带货重写、短句口播化。
- 知识库：脚本结构、表达重组、痛点重写、不同风格话术。
- 输出：新文案版本、改写逻辑、与原文差异、可拍摄建议、合规提醒。

带货完整骨架：

- 输入：产品/服务、价格、目标人群、购买场景、卖点、证明材料、成交入口、平台。
- 层级选项：产品需求、成交理由、信任证明、商品视觉化、顾虑化解、CTA、商品卡/小黄车/直播间/TikTok Shop、数据复盘。
- 知识库：带货视频需求拆解、成交理由、信任证明、商品视觉化、TikTok补充、复盘表。
- 输出：产品需求拆解、目标人群和购买情境、成交理由、信任证明、带货脚本、商品视觉化拍摄清单、承接话术、CTR/CVR/GMV复盘动作。

### 脚本创作模块

脚本创作必须采用层级选择，保持原站「点开继续选择」体验：

```text
脚本创作
-> 个人IP脚本 / 成交脚本 / 带货脚本
-> 子类型
-> 填写输入
-> 生成结果
```

个人IP脚本子类型：

- 教知识：信息差、步骤、判断标准、收藏价值。
- 晒过程：服务现场、交付过程、证明材料、信任建立。
- 讲故事：客户案例、转折、结果、价值升华。
- 聊观点：立场、争议、反常识、评论互动。

带货脚本子类型：

- 种草：场景需求、兴趣激发、产品引入。
- 测评：横向测评、纵向测评、极限测评、替用户质检。
- 过程：制作、质检、服务、门店、工厂。
- 故事：客户处境、错误选择、解决过程、结果变化。
- 直播预热：直播获得感、福利、当下问题、进房理由。

### 其他模块层级

IP定位：

```text
IP定位
-> 生成采集表
-> 用户填写行业、产品/服务、价格、目标客户、信任证明、出镜能力、成交入口
-> 后台读取个人IP知识库定位流程
-> 输出IP定位一句话、商业路径、用户画像、人设资产、内容矩阵方向和待确认项
```

IP定位采集表不是只问少量问题，而是一个分组表单。首屏先展示关键必填项，用户可展开填写高级项。

基础信息：

| 字段 | 必填 | 用途 |
|---|---|---|
| 行业/赛道 | 是 | 判断行业语境和内容边界 |
| 身份/角色 | 是 | 判断账号属于专家、老板、老师、顾问、服务商、创始人还是产品型账号 |
| 从业年限/经历 | 否 | 提炼经验背书和人设资产 |
| 所在城市/服务范围 | 否 | 判断本地到店、全国服务或跨境电商场景 |
| 账号现状 | 否 | 新号、老号、已有粉丝、已有成交、已有内容库 |

商业目标：

| 字段 | 必填 | 用途 |
|---|---|---|
| 产品或服务 | 是 | 明确最终变现对象 |
| 客单价/价格区间 | 是 | 判断成交难度、信任成本和内容承接方式 |
| 主要变现方式 | 是 | 咨询、服务、课程、实体到店、直播、私域、带货、TikTok Shop |
| 最想卖给谁 | 是 | 明确核心客户，不做泛人群 |
| 当前最想解决的问题 | 否 | 起号、涨粉、咨询、成交、复购、直播转化或商品点击 |

目标用户：

| 字段 | 必填 | 用途 |
|---|---|---|
| 目标客户身份 | 是 | 建立人群印记 |
| 当前阶段/具体场景 | 是 | 生成场景化选题和脚本开头 |
| 核心痛点 | 是 | 后续生成痛点选题、脚本钩子和成交链路 |
| 潜在需求 | 否 | 挖掘用户还没说出口但会被击中的需求 |
| 购买/咨询顾虑 | 否 | 用于信任证明、顾虑化解和成交话术 |

人设资产：

| 字段 | 必填 | 用途 |
|---|---|---|
| 资质/证书/头衔 | 否 | 作为专业背书 |
| 案例/结果证明 | 否 | 作为信任证明和故事素材 |
| 客户评价/口碑 | 否 | 作为第三方证明 |
| 个人经历/转折故事 | 否 | 用于讲故事脚本和人设记忆点 |
| 价值观/反对什么 | 否 | 用于聊观点、筛选精准粉和建立立场 |

内容条件：

| 字段 | 必填 | 用途 |
|---|---|---|
| 是否真人出镜 | 是 | 决定脚本表达和拍摄方式 |
| 是否能直播 | 否 | 判断直播承接和预热内容 |
| 是否能展示过程/现场/材料 | 否 | 决定晒过程、商品视觉化和信任证明 |
| 可用素材 | 否 | 客户案例、产品图、门店、工厂、评价、聊天记录、前后对比 |
| 风格偏好 | 否 | 专业、温和、犀利、克制、幽默、朴实、热血等 |

承接与合规：

| 字段 | 必填 | 用途 |
|---|---|---|
| 成交入口 | 是 | 评论、私信、表单、到店、直播、小黄车、私域或 TikTok Shop |
| CTA偏好 | 否 | 评论关键词、私信话术、预约、进直播、点商品卡 |
| 禁忌/合规边界 | 否 | 避免夸大承诺和高风险表达 |
| 不能提的内容 | 否 | 品牌限制、平台规则、敏感词、行业合规边界 |

## 知识库架构

首版使用项目内文件知识库：

```text
knowledge/
  handbooks/
    personal-ip.md
    commerce-video.md
    combined.md
  templates/
    ip-plan.schema.json
    commerce-plan.schema.json
    script.schema.json
    review.schema.json
  manifest.json
```

原则：

- `knowledge/` 随项目一起部署到 VPS。
- 服务端只读取项目内知识库文件。
- 不依赖 `C:\Users\...` 等本地绝对路径。
- `manifest.json` 记录知识库版本、文件 hash、模块分类和更新时间。
- 后续可以升级为 SQLite 索引或向量检索，但首版不引入这些复杂依赖。

## 模块与知识库映射

每个内部模块必须提前绑定知识库内容、提示词模板和输出结构。前端不能只把用户输入直接发给模型。

| 模块 | 任务类型 | 主要知识库内容 | 输出重点 |
|---|---|---|---|
| IP定位 | `personal_ip` 或 `combined` | 个人IP核心公式、商业定位、目标用户、人设资产、内容矩阵、默认交付包 | IP定位一句话、采集表、商业路径、用户画像、人设资产、待确认项 |
| 爆款选题 | `personal_ip` 或 `combined` | 个人IP手册中的选题系统、8类爆款元素、人群/场景/痛点/情绪公式 | 爆款元素分类、选题标题、适用人群、拍摄角度 |
| 成交选题 | `personal_ip` 或 `combined` | 个人IP商业定位、目标用户、信任资产、咨询/私域/直播承接 | 成交型选题、信任证明、CTA、承接路径 |
| 痛点选题 | `personal_ip` 或 `commerce_video` | 目标用户画像、需求场景、痛点、顾虑、购买冲突 | 痛点清单、场景选题、情绪钩子 |
| 脚本创作 | `personal_ip`、`commerce_video` 或 `combined` | 个人IP四类脚本卡、带货成交链路、内容类型脚本结构 | 完整口播脚本、镜头提示、CTA |
| 文案二创 | `personal_ip` 或 `commerce_video` | 选题系统、脚本结构、平台表达方式、差异化表达规则 | 二创版本、口播优化、标题/封面文案 |
| 爆款拆解 | `personal_ip` 或 `commerce_video` | 爆款结构、黄金3秒、情绪刺点、成交链路、商品视觉化 | 拆解报告、可复用结构、下一条仿写方向 |
| 文案洗稿 | `personal_ip` 或 `commerce_video` | 脚本结构、表达重组、痛点重写、不同风格话术 | 新文案版本、结构变化说明、风险提醒 |
| 带货 | `commerce_video` 或 `combined` | 带货视频手册中的需求拆解、成交理由、信任证明、商品视觉化、TikTok补充、复盘表 | 带货链路、带货脚本、商品卡/小黄车/直播承接、CTR/CVR/GMV复盘 |

每个模块的 prompt 至少包含：

1. 模块专用任务说明。
2. 对应知识库片段。
3. 用户输入。
4. 输出结构约束。
5. 合规和事实核验提醒，尤其是产品功效、价格、资质、医疗健康、金融法律等高风险内容。

## 后端架构

建议使用 React + Vite 前端，Node.js 服务端。后端模块：

```text
server/
  config/
    env.ts
  knowledge/
    loadKnowledge.ts
    manifest.ts
  prompt-engine/
    routeTask.ts
    buildPrompt.ts
    schemas.ts
  providers/
    openaiCompatible.ts
  routes/
    generate.ts
```

### routeTask

根据用户输入和当前模块判断任务类型：

- `personal_ip`
- `commerce_video`
- `combined`

判断逻辑：

- 专家、老板、老师、顾问、服务商、创始人、个人账号：优先个人IP。
- 产品、TikTok Shop、小黄车、直播、商品卡、门店成交：优先带货视频。
- IP账号同时卖产品或服务：使用 combined。

### loadKnowledge

按任务类型读取知识库：

- `personal_ip` 读取 `personal-ip.md` 和相关模板。
- `commerce_video` 读取 `commerce-video.md` 和相关模板。
- `combined` 读取两份手册，并追加组合任务规则。

### buildPrompt

每次请求组装四层内容：

1. 系统角色和输出约束。
2. 知识库方法论片段。
3. 用户输入和当前模块选择。
4. 结构化输出格式要求。

### openaiCompatible

通过环境变量调用兼容 OpenAI API 的接口：

```env
OPENAI_BASE_URL=https://api.example.com/v1
OPENAI_API_KEY=your_key
OPENAI_MODEL=model-name
```

API Key 只保存在服务端 `.env`，前端不可见。

## API 设置与访问方式

首版面向个人 VPS 或小团队自用，不做注册登录。用户只需要配置兼容 OpenAI API 的模型接口即可使用。最稳定的 API 配置方式是服务端环境变量：

```env
OPENAI_BASE_URL=https://api.example.com/v1
OPENAI_API_KEY=your_key
OPENAI_MODEL=model-name
```

推荐策略：

- 默认用 `.env` 配置模型接口，部署脚本负责提示用户填写。
- 前端不暴露 API Key，不允许普通访问者输入或查看 API Key。
- 首版不做用户注册、登录、账号体系和权限体系。
- 首版可以提供一个本地/部署后的「API设置」页面，但它只负责写入或提示更新服务端模型配置，不是用户中心。
- 如果 API 设置页在公网可访问，必须用部署脚本生成的 `ADMIN_SETUP_TOKEN` 或一次性初始化口令保护。
- 多用户注册、套餐、用量计费、每个用户自带 API Key 等能力放到后续版本。

API设置页首版可包含：

- API Base URL。
- 模型名。
- API Key 更新输入框，默认不回显已保存 key。
- 连接测试按钮。
- 当前知识库版本和 manifest 自检结果。
- 保存后只写入服务端安全配置文件或提示管理员更新 `.env`，不能写入前端文件。
- API 配置完成后，工作台无需登录即可使用。

## VPS 部署

首版提供一键部署脚本，目标为常见 Ubuntu VPS。

部署内容：

- 安装 Node.js 运行环境。
- 拉取或复制项目文件。
- 安装依赖。
- 检查 `knowledge/manifest.json`。
- 生成 `.env` 示例。
- 构建前端。
- 启动后端服务。

首版不强制绑定域名和 HTTPS，但如果开启 API 设置页或公网访问，部署脚本必须提示配置 HTTPS、初始化口令和防火墙规则。

## 验证标准

功能验证：

- 能在前端选择原版命名的内部模块。
- 脚本创作模块能逐级展开选择。
- 带货模块能作为单独模块进入。
- 用户输入能提交到后端。
- 后端能读取 `knowledge/`。
- 后端能按模块组装 prompt。
- 能调用兼容 OpenAI API 的模型。
- 能展示结构化结果。

视觉验证：

- 工作台结构接近参考站：深色背景、外层应用栏、内部模块导航、右侧内容、顶部标题。
- 除外层应用栏外，内部工具选项名称尽量和原版一致。
- 脚本创作选择保留原站式层级展开体验。
- 移动端不要求完全复刻原站，但必须可用，不出现文字重叠。

部署验证：

- VPS 脚本能完成依赖安装、构建和服务启动。
- `.env` 缺失时有明确提示。
- `knowledge/manifest.json` 缺失或 hash 不匹配时有明确错误。

## 非目标

- 首版不做多人账号权限。
